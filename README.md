# Flickr8k Explorer

A local tool for exploring the [Flickr8k](https://huggingface.co/datasets/jxie/flickr8k) dataset
(8,000 images, 5 captions each): browse the samples, search them by meaning, inspect any example in
detail, and see the shape of the whole dataset on a 2D map of its embedding space.

Everything runs on one machine. No cloud services, no managed database, no paid APIs.

| View | What it answers |
| --- | --- |
| **Browse** | What is in this dataset? Grid of samples, filterable by split. |
| **Search** | Where are the images of *a dog jumping over a fence*? CLIP semantic search, or BM25 over the captions. |
| **Detail** | What does one example look like? Full image, all five captions, resolution and file metadata, and its nearest neighbours in embedding space. |
| **Map** | How is the dataset distributed? UMAP projection of the CLIP embeddings, coloured by cluster. Search hits are highlighted in place. |
| **Stats** | What is the dataset made of? Split sizes, caption-length distribution, resolutions, and the most frequent caption words. |

## Requirements

- Python 3.11 or 3.12, via [uv](https://docs.astral.sh/uv/) (`curl -LsSf https://astral.sh/uv/install.sh | sh`)
- Node.js 18+
- About 2.5 GB of free disk space and an internet connection for the first run (dataset + model download)

## Setup

```bash
git clone <this-repo> && cd Mobileye
uv sync --extra dev          # Python environment
npm --prefix frontend install
```

## 1. Ingest the dataset

```bash
uv run python -m backend.ingest
```

This is a one-time step, roughly **8 to 12 minutes** on an Apple Silicon laptop. It:

1. downloads the parquet shards from HuggingFace (~275 MB, cached under `~/.cache/huggingface`);
2. writes the images and 320 px thumbnails to `data/images` and `data/thumbs`, and fills
   `data/flickr8k.db` (images, captions, and an FTS5 index over the captions);
3. embeds every image with CLIP ViT-B/32 into `data/embeddings.npy` (~16 MB);
4. projects those embeddings to 2D with UMAP and labels clusters with KMeans.

**To try it quickly**, ingest a subset instead: `uv run python -m backend.ingest --limit 200`
(200 images per split, under a minute after the download).

Useful flags:

| Flag | Effect |
| --- | --- |
| `--limit N` | Only ingest N images per split |
| `--skip-embeddings` | Browse and caption search only, no CLIP |
| `--skip-projection` | Skip the UMAP map |
| `--only {decode,embed,project}` | Re-run a single stage against the existing database |

Re-running the ingestion is safe: it rebuilds the tables from scratch, and both the dataset download
and the model weights are cached.

## 2. Run the application

Two terminals:

```bash
# terminal 1 - API on http://127.0.0.1:8000 (docs at /docs)
uv run uvicorn backend.app.api:app --reload

# terminal 2 - UI on http://localhost:5173
npm --prefix frontend run dev
```

Open <http://localhost:5173>. The Vite dev server proxies `/api` and `/media` to the backend.

The first semantic search loads the CLIP text encoder and takes a few seconds; every search after
that is instant. Browsing, caption search and stats never load the model.

### Single-process alternative

```bash
npm --prefix frontend run build
uv run uvicorn backend.app.api:app     # serves the built UI and the API on :8000
```

## Tests

```bash
uv run pytest
```

The suite builds a miniature dataset on disk and exercises the HTTP API against it, with a stub
embedder, so it needs neither the real dataset nor the model weights.

## Project layout

```
backend/
  ingest.py           one-shot pipeline: parquet -> images + SQLite + vectors + projection
  app/
    api.py            FastAPI routes
    services.py       application state, search logic, lazy model loading
    repository.py     every SQL statement
    embeddings.py     Embedder interface, CLIP implementation, vector index
    db.py             schema and connection helpers
    models.py         response schemas (the frontend contract)
    config.py         settings, overridable via F8K_* environment variables
  tests/
frontend/src/
  App.tsx             view/query state
  api.ts              typed client
  hooks.ts            data-fetching hooks
  components/         ImageGrid, SearchBar, DetailPanel, ProjectionMap, StatsPanel
data/                 generated by the ingestion, gitignored
```

## API

| Endpoint | Purpose |
| --- | --- |
| `GET /api/dataset` | Counts, splits, embedding model in use |
| `GET /api/images?split=&offset=&limit=` | Paginated browse |
| `GET /api/images/{id}` | Full detail with all captions |
| `GET /api/images/{id}/similar?limit=` | Nearest neighbours in embedding space |
| `GET /api/search?q=&mode=semantic\|text&split=&limit=` | Ranked search |
| `GET /api/projection?split=` | 2D coordinates and cluster labels |
| `GET /api/stats` | Dataset composition |

Interactive documentation is at <http://127.0.0.1:8000/docs>.

## Configuration

Any setting in `backend/app/config.py` can be overridden with an `F8K_`-prefixed environment
variable or a `.env` file, for example:

```bash
F8K_CLIP_MODEL=ViT-L-14 F8K_CLIP_PRETRAINED=laion2b_s32b_b82k uv run python -m backend.ingest
```

## Troubleshooting

**"Backend unavailable" in the UI, or a 503 from the API.** The dataset has not been ingested yet.
Run step 1.

**Semantic search is disabled.** The database has no embeddings. Re-run the ingestion without
`--skip-embeddings`, or just the one stage: `uv run python -m backend.ingest --only embed`.

**The map view says no projection is stored.** Run `uv run python -m backend.ingest --only project`.
